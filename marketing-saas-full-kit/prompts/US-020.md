## Implementation instructions

1. **No new DB tables** — Attribution is computed on-demand from `metricRecord`. This is a pure analytical endpoint.

2. **Tests first** — One test per AC:
   - AC1: GET /api/attribution?model=last_click → returns channel contribution data
   - AC2: Switching model → different distribution returned (test that linear gives equal shares, last_click gives 100% to last)
   - AC3: Table response includes directConversions, assistedConversions, attributedRevenue, sharePercent per platform
   - AC4: Only one platform connected → API returns 400 with explanation message
   - AC5: Tooltip data included in response per platform (same fields as table)
   - AC6: Platform with no conversion data → hasConversionData: false, conversions shown as null not 0

3. **Attribution models to implement:**
   - `last_click` — 100% credit to the platform with the most recent conversion event. Since we don't have session-level data, approximate using: platform with highest conversion count in period gets full credit.
   - `first_click` — 100% credit to the platform with the earliest conversion events in the period.
   - `linear` — Equal credit split across all connected platforms that had any conversions.
   - `time_decay` — Platforms with more recent conversion activity get more credit. Weight = (days since period start) / total days. Normalize to sum to 100%.

4. **Data availability notice** — Always include in API response:
```json
{ "dataNote": "Attribution is based on platform-reported conversion data, not session-level tracking. Results reflect each platform's self-reported conversions and may overlap." }
```
This note must be displayed in the UI.
