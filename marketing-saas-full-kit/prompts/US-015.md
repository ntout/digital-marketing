## Implementation instructions

1. **Database schema** — Add `AiApprovalQueueItem` model using the exact fields in CONTEXT.md.

2. **Tests first** — One test per AC:
   - AC1: AI proposes action with permission='approval' → queue item created, in-app notification triggered
   - AC2: GET /api/ai/approval-queue → returns pending items with all required fields
   - AC3: POST /api/ai/approval-queue/:id/approve → platform API called, change log entry created (actorType: 'ai'), item status = 'approved'
   - AC4: POST /api/ai/approval-queue/:id/reject → no platform call, item status = 'rejected', rejection logged
   - AC5: BullMQ job checks items older than 48hrs → status = 'expired', manager notified
   - AC6: Kill switch active → approve endpoint returns 423 with explanation

3. **Queue item expiry job** — Add a BullMQ repeatable job that runs every hour, finds pending items older than 48 hours, marks them expired, and sends notifications.

4. **Connection to AI agent jobs** — AI agent jobs (US-016+) will call a shared `AiActionDispatcher` service. Create this stub now:
```ts
// apps/web/src/lib/services/AiActionDispatcher.ts
AiActionDispatcher.dispatch(workspaceId, actionType, payload, reason) => Promise<void>
// Internally: calls AiGuardrails.check(), then routes to queue, executes, or logs suggestion
```
