We are implementing **US-004: Background data ingestion & sync**.

Read the full story in `stories/US-004-data-ingestion.md`.

## Instructions

1. **Database schema** — Add `MetricRecord` and `SyncLog` models. Use the exact fields in CONTEXT.md. Upsert key: `(workspaceId, platform, campaignId, adSetId, date)`.

2. **Job queue setup** — Configure BullMQ with Redis. Create two job types:
   - `backfill` — triggered once on new connection, fetches last 90 days
   - `sync` — scheduled every 4 hours per workspace per platform, fetches last 3 days

3. **Tests first:**
   - AC1: Creating a PlatformConnection enqueues a backfill job
   - AC2: Backfill job normalizes API response and upserts into metricRecord
   - AC3: Sync job upserts only the last 3 days, updating existing records
   - AC4: Rate limit error → job re-queued with backoff (5min, 15min, 60min)
   - AC5: 3 consecutive failures → connection status = 'reconnect_required', notification sent
   - AC6: GET /api/connections → includes lastSyncedAt and sync status per connection

4. **Data normalization** — Each platform returns different field names. Create a normalizer per platform that maps to the unified `MetricRecord` shape. Put these in `src/normalizers/`.
