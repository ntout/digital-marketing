## Why this story is critical
Every AI autonomous action, approval queue resolution, and revert in all future stories writes to `campaignChangeLog`. This story extends that model and builds the Activity Log UI. If the schema or revert logic is wrong here, all AI stories will break.

## Implementation instructions

1. **Database schema** — Extend `CampaignChangeLog` with AI-specific fields (see CONTEXT.md and story file): `actorType`, `aiReason`, `aiTriggeredBy`, `revertedAt`, `revertedByLogId`. Migrate existing records: set `actorType = 'human'` for all existing entries.

2. **Tests first** — One test per AC:
   - AC1: Change log entry created with all fields for both human and AI actors
   - AC2: GET /api/activity-log → paginated, filterable by actorType, platform, campaign, date
   - AC3: AI log entries include aiReason and aiTriggeredBy fields in response
   - AC4: POST /api/activity-log/:id/revert → confirmation modal shows before/after, on confirm re-applies old value via platform API
   - AC5: Revert success → original entry marked revertedAt, new entry created with reference to original
   - AC6: Viewer → revert button not rendered, revert endpoint returns 403

3. **Revert logic** — The revert action is itself a write-back call through `PlatformApiService`. It creates a NEW `campaignChangeLog` entry (actorType: 'human') with `oldValue` = current platform value and `newValue` = the reverted-to value. It then updates the original entry's `revertedAt` and `revertedByLogId`.

4. **AiGuardrails utility** — Create `src/utils/AiGuardrails.ts` as a stub now. It will be fully implemented in US-014 but needs to exist so US-015+ can import it. Stub it to return `{ allowed: true }` for all checks.
